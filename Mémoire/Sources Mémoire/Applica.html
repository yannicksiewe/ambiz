Application mobile native ou hybride : il faut choisir !

Application mobile native ou hybride : il faut choisir !

18/04/2013 12:15 1 commentaire

Chez ButterflyEffect, nous développons exclusivement des applications mobiles hybrides. Pourquoi ce choix ? Vous le saurez en lisant cet article qui vous rendra incollables sur les avantages et les inconvénients de ce type d’applications.

 

Les bases

Tout comme votre ordinateur peut être un PC ou un Mac, votre smartphone peut également être un iPhone, un Android, un Windows Phone, ou un BlackBerry. Les différences entre ces différents téléphones ne sont pas qu’esthétiques : elles sont surtout techniques ! Ce qui signifie qu’un logiciel (ce qui dans le cas des smartphones est appelé application) réalisé spécifiquement pour un iPhone ne pourra être exécuté par un Android, et vice-versa. Le choix des plateformes sur lesquelles vous souhaiter que votre application mobile puisse s’exécuter est donc crucial.

 

Les applications natives 

La façon la plus évidente de développer une application pour une plateforme donnée est de l’écrire en code natif, ce qu’on pourrait appeler le langage par défaut du smartphone. Par exemple, le langage natif de l’iPhone est l’Objective-C, celui d’Android est le Java.

Le développement en code natif présente de nombreux avantages : c’est lui qui permet d’avoir les meilleures performances, la meilleure gestion des fonctionnalités avancées du téléphone, et donc de garantir la meilleure expérience utilisateur. Le développement d’un jeu en 3D ou une application de réalité augmentée seront forcément basés sur du développement en code natif.

Evidemment, il présente également des inconvénients, le principal étant qu’une application écrite spécifiquement (c’est-à-dire en code natif) pour iPhone ne fonctionnera pas sous Android. Pour que votre application soit disponible sur ces deux plateformes, les coûts dans le projet affectés au développement seront donc multipliés par deux (et donc par quatre si vous voulez que votre application soit disponible sur les quatre plateformes principales) ! Ce qui n’est pas à la portée de tout le monde !

 

Les applications hybrides

Ne vous inquiétez pas, il existe une alternative : les applications hybrides.  Ces applications sont basées sur un ensemble de langages communs entre toutes les plateformes. Dans ce cas, les plus souvent utilisés sont les langages issus du Web : HTML, CSS, JavaScript.

Une application hybride, qu’elle soit destinée à Android ou un iPhone, sera donc construite sur ces technologies standards, connues et exécutées par tous les smartphones du marché. Nous perdons donc la dépendance à une plateforme spécifique, pour un code exécutable partout ! Vous l’aurez compris, cela induit une réduction du temps (et donc des coûts) liée au développement ainsi qu’une facilité de maintenance. Il est également à noter que certaines portions sont réalisées en code natif pour accéder aux fonctions avancées du téléphone (notifications Push par exemple), rendues accessibles par des « ponts » entre la partie native et la partie hybride. Le développement hybride tire donc partie du meilleur des deux mondes.

Bien évidemment, le développement hybride a lui aussi ses inconvénients, le principal résidant dans les performances de l’application réalisée, particulièrement pour les smartphones anciens (iPhone 3GS et inférieurs, Android < 4, etc.).

Cependant, un utilisateur disposant d’un téléphone récent ne verra probablement pas la différence avec une application native, si votre application ne nécessite pas trop de ressources. En effet, l’application hybride sera disponible sur les marchés d’application (App Store, Play Store), s’affichera sous forme d’une icône sur son « bureau » (appelé springboard sur iPhone) et pourra recevoir des notifications.

 

Conclusion 

Si votre budget ne permet pas la réalisation d’une application native ou que vous souhaitez être accessible plus rapidement sur tous les smartphones du marché, une application hybride est faite pour vous !

De grandes sociétés ont déjà fait le choix de l’hybride (LinkedIn, le Financial Times pour ne citer qu’eux) ce qui confirme la pertinence de ce système. Enfin, le groupe Gartner a récemment déclaré qu’en 2016, 50% des applications seraient hybrides (source).

Et vous, pensez-vous que le développement hybride soit une bonne solution ?
